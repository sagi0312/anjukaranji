###Alright, enough Force philosophy. Let's talk about what Obi-Wan is actually doing in that desert.

With every debounced keystroke, ESLint quietly parses your JavaScript into this **AST**. Basically, it takes your itty-bitty
`const x = 1`
and turns it into AST (Nodes. Relationships. Structure… a complete ancestry report)... A whole crime scene! (Over one variable. Iconic!😏)

_Don't be a dick, Anju. Explain the code!_

Before ESLint starts walking the AST, ESLint's already mapped all of your codebase. It builds a scope tree - basically tracking which variables live where, and who can access what.

Then each rule registers which node types it cares about and builds the listener map.

During traversal, the listener map decides which rules run, and those rules use the scope tree to decide whether to call `report()`.

Think of it as:

> If a `CallExpression`(Sith) appears… activate only the assigned Jedis(RuleA, RuleB) (It's almost corporate).
> And then the Jedi analyze the situation… and decide whether to draw their sabers.

```javascript
{
  CallExpression: [ruleA, ruleB],
  Identifier: [ruleC],
  VariableDeclaration: [ruleD]
}
```

This is why ESLint scales

Now, for some Leetcode. Remember DFS? Visitor pattern? That's the traversal algorithm used here.

###Now… baby Yoda time

Today, we are not recreating ESLint. Obi-Wan has seen wars. We are building the bite-sized baby Yoda.

Our force can catch every rogue `console.log` left behind in production code.

Here's the thing about our Grogu... he’s not the sharpest tool in the shed.
Basically: see the frog → eat the frog.

It visits every single node in the AST and calls `report()`.

```javascript
function walk(node, visitor) {
  if (!node || typeof node !== "object") return;

  if (visitor[node.type]) {
    visitor[node.type](node);
  }

  for (const key of Object.keys(node)) {
    walk(node[key], visitor);
  }
}

walk(ast, {
  CallExpression(node) {
    noConsoleLog(node, report);
  },
});
```

But the Big Bro (ESLint) can optionally attach a fix descriptor like, "_delete characters 42 through 67, replace with this string._". Pass `--fix` and it collects all the patches and applies them.

That’s it! Then we wrap it up as a CLI tool using `npm link`, point it at any repo, and let it chase the frogs.

###The new kids on the block

Biome and Oxlint are written in Rust and make ESLint seem like a boomer.

On a 1,000-file repo, ESLint might take 40 seconds. oxlint does it in under 2.

The reason is straight-forward. ESLint is JavaScript linting JavaScript. So, single-threaded. Rust linters are multi-threaded.

Worth a look if you're starting fresh. Just know that the plugin ecosystem is still catching up. Obi-Wan is a middle-aged Jedi after all.
