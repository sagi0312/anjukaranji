## Now, let's talk about Obi-Wan's Job Description.

In VS Code, ESLint parses every debounced keystroke and converts JavaScript into AST. It takes your tiny little `const x = 1` and turns it into AST (Nodes. Relationships. Structure... a complete ancestry report) - A whole crime scene! Over 1 variable. Iconic! 😏

Before it starts walking the AST, it's already mapped your codebase and built a scope tree (basically tracking which variables live where, and who can access what).

Then each rule registers for the node type it cares about. And voila! The listener map is ready.

During the traversal, the listener map decides which rules to apply, and those rules use the scope tree to decide whether to call `report()`.

Think of it as:

> If a CallExpression appears, activate only RuleA. Joker walks in, only Batman stands up.
```javascript
{
  CallExpression: [ruleA],
  Identifier: [ruleC],
  VariableDeclaration: [ruleD]
}
```

You might already know this part — remember Leetcode? DFS? Visitor pattern? That's the traversal algorithm ESLint uses.

(Your little `const x = 1` you ask... triggers nothing.)
ESLint walks past it like: Not today baby!

## Enter Grogu.

Today, let's make a baby-sized linter... for the heck of it. Not Obi-Wan. He's seen wars. But a bite-sized baby Yoda.

Our force can catch every rogue `console.log` left behind in code.
Here's the thing about Grogu though... he's not the sharpest tool in the shed.
Grogu logic: see the frog → eat the frog.

It visits every single node in the AST and calls `report()`.

Here's the whole traversal - super simple:
```javascript
// The traversal accepts AST and the map only bothers itself with 'CallExpression' node.
// The only Jedi assigned to that node is 'noConsoleLog'.

walk(ast, {
  CallExpression(node) {
    noConsoleLog(node, report);
  },
});

function walk(node, visitor) {
  // Base case: if this isn't a real AST node, return early
  if (!node || typeof node !== "object") return;

  // If we have a Jedi registered for this node type, fire it
  if (visitor[node.type]) {
    visitor[node.type](node);
  }

  // Recurse through all its children just to be sure
  for (const key of Object.keys(node)) {
    walk(node[key], visitor);
  }
}
```

Grogu only tattles. ESLint actually cleans up after you. Pass `--fix` and it gathers all the fixes and applies them at the end.
`npm link` makes it globally accessible on your machine.
That's it. Your very own prickly little conscience. Ours just has one hill it'll die on: `console.log`.

## The new kids on the block

Word travels fast 'round these parts. And the word is, there's new kids in town.

Biome and Oxlint are written in Rust. On a 1,000-file repo, ESLint might take 40 seconds. Oxlint does it in under 2. ESLint is JavaScript. Single-threaded by nature.

These new plugin ecosystems are however still catching up. I tried Biome on a personal project and liked it. Worth your time — YouTube has solid tutorials if you want to go deeper. [I like this one.](https://www.youtube.com/watch?v=x5kwmIaPty8&t=48s)

ESLint *did* hold the fort down for 13 years. That kind of time does help you load up on ammunition.
